Ideas behind Proximodo

The main idea is to provide a free and open alternative to the excellent but alas now unmaintained Proxomitron. I hope that in a near future Proximodo will implement all functionalities of Proxomitron 4.5, and more. Currently, 100% of Proxomitron $commands and syntax are supported, so you can use all your custom filters with Proximodo!

Also, the first version (0.1) of Proximodo already offers new functionalities, such as:

  • Multilingual GUI
  • Multiplatform (thanks to wxWidgets and no use of Windows specific libraries)
  • Centralized bank of filter: Proxomitron's .cfg files are replaced by mere enumerations of active filters, i.e "Configurations"
  • Export/Import filters from/to clipboard (no more need to mess with .txt files)
  • Import from Proxomitron filters (again through clipboard)

Missing options still to be developed

  • HTML debug info (but do we want it?)
  • SSL support (https)

Known issues

  • While gzip/deflate output works most of the time, there seem to be a few pages that get corrupted when compressed. In this case the browser asks the website for the missing pieces, and the latter returns a 206 response that is not correctly appended into the browser. Example: http://www.calgarytransit.com/ and create a filter force incoming header Content-Encoding to gzip (or deflate).
  • In some cases, web page retrieval does not behave the same when using a next proxy or not. Example: groups.yahoo.com will return a 302 response with an invalid Location header, which is not the case when the browser connects directly to the internet or through another proxy server.
  • There is a lot of flickering when resizing the window. It is a known issue of wxWidgets. I have not yet figured out a way to get rid of it without loosing in code clarity.
  • There are some redrawing glitches in the GUI, again on resize. I think they should disappear with the next stable release of wxMSW (release 2.5.3 is still under development.)

Framework

I chose wxWidgets for GUI and sockets. I used wxMSW 2.5.3 for developing v0.1, so due to backward compatibility issues the code may not compile or work properly with earlier versions of wxWidgets (for example, earlier versions won't display the popup menu of the wxTaskBar).

I use C++ STL wherever possible (as long as it does not make code unreadable). In particular, string is used instead of char[] as much as possible.

Proximodo does not use standard C libraries, only pure C++.

I did not compile wxMSW with STL wxString inheritance, I am not sure it would have been better, but as for now there are .c_str() calls at every conversions between wxString and string.

Other notes on development choices

Case sensitivity: Header names and $SET variable names are case-insensitive. List names and configuration names are case-sensitive. In pattern, everything is case-insensitive except [%xx] and [\letter].

Want to participate?

If you are willing to participate in Proximodo's development, you are very welcome. Parts that most need help:

  • Testing (engine/proxy/GUI)
  • Debugging and tracking memory leaks (if any)
  • Completing help pages
  • Translating help pages and .lng files to other languages
  • Porting to other platforms (I don't have Linux installed right now, but a Knoppix distro)
  • Creating a nice website
  • Developping new or missing options

All your suggestions, encouragements or even flaming will be welcome!
